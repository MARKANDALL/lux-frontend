## Top Investigation
If you have 15 minutes today, look at `features/voice-mirror/voice-mirror.js:606-612` because the player div is removed on every remount without revoking its blob URL or pausing the audio — leaks one Object URL per remount and can leave previous synthesis audio playing.

## Findings

### CRITICAL

**`features/voice-mirror/voice-mirror.js:606-612`** — Blob URL leak + audio left playing on remount.

`mountVoiceMirrorButton` strips an existing `.lux-voice-mirror-player` via `oldPlayer.remove()` but never calls `URL.revokeObjectURL(oldAudio.src)` (the way `showPlayer` does at line 518) and never pauses the underlying `<audio>`. Each remount of the voice mirror (e.g., re-mounting after voice-onboarding completes, or any caller calling `mountVoiceMirrorButton` twice) leaks a blob URL holding the entire synthesized audio buffer. In some browsers (notably Safari/WebKit), removing an `<audio>` element from the DOM while it is playing does not synchronously stop playback, so the user can also end up hearing the previous mirror over the new one.

```js
// before
const oldPlayer = container.querySelector('.lux-voice-mirror-player');
if (oldPlayer) oldPlayer.remove();

// after
const oldPlayer = container.querySelector('.lux-voice-mirror-player');
if (oldPlayer) {
  const oldAudio = oldPlayer.querySelector('audio');
  if (oldAudio) {
    try { oldAudio.pause(); } catch {}
    if (oldAudio.src) URL.revokeObjectURL(oldAudio.src);
  }
  oldPlayer.remove();
}
```

Confidence: high on the blob-URL leak; medium on the dangling-audio symptom (depends on browser).

### WARNING

- `features/voice-mirror/voice-mirror.js:514-520` — `showPlayer` revokes the old `audio.src` and removes the player, but does not call `audio.pause()` first; rapid back-to-back synthesis can briefly overlap audio on engines that don't stop playback on detach.
- `features/voice-mirror/voice-mirror.js:596-633` — `mountVoiceMirrorButton` is async with no in-flight guard. Two rapid calls interleave as `remove → await → append`, so the second call's `remove` runs before the first call's `append`, and the container ends up with two shells. Caller (`voice-onboarding` completion in `createSetupButton`) plus an external trigger could realistically race here.
- `features/features/selfpb/dom.js:240-252` — `vmSlotHome` / `vmSlotNext` are captured only on the first move into `spb-vmMount`. If `mountTTSPlayer` re-runs while expanded is open (it falls back to a new `#tts-voice-mirror-slot` because `host.querySelector` cannot see the slot now living in the float), you end up with the original slot orphaned in `spb-vmMount` and a brand-new slot in the TTS host that closeExpanded never restores anywhere.

### INFO
