## Top Investigation
If you have 15 minutes today, look at `ui/auth-dom.js` because the `signInWithOtp` call and `onAuthStateChange` handler have no in-flight guards or throw-path recovery, so a flaky network or a double auth event can wedge the login button or fire migration twice.

## Findings

### CRITICAL
✅ None

### WARNING

- `ui/auth-dom.js:102-111` — `await supabase.auth.signInWithOtp(...)` is only checked via the `{ error }` return shape; if the call *throws* (network blip, CORS failure, uncaught SDK error), the button is left disabled with "Sending..." text forever, with no try/catch to restore it. The error path (line 107) only handles resolved errors.
- `ui/auth-dom.js:132-165` — `onAuthStateChange` is registered with an async callback and has no in-flight guard. If Supabase fires `SIGNED_IN` (or `INITIAL_SESSION` then `SIGNED_IN`) before the first `await migrateHistory(...)` completes, `setUID(realUid)` hasn't run yet, so the second invocation still sees `guestUid !== realUid` and fires migration a second time for the same guest. Server-side idempotency unknown; worth verifying.

### INFO

- `ui/auth-dom.js:161-165` — `setUID(realUid)` runs even when `migrateHistory` silently fails (catch on line 196 swallows). User ends up logged in with potentially missing history and no visible warning. Possibly intentional, but worth a surface UI hint.
- `ui/ui-ai-ai-dom.js:215-222` — `document.getElementById('en-block-${idx}')` uses an unscoped global id. Safe today because `contentArea.innerHTML` is overwritten per render, but brittle if another panel ever uses the same `en-block-N` pattern.
- `ui/ui-ai-ai-dom.js:96-108` — The `cloneNode(true) + replaceChild` trick to drop old listeners rebuilds the button each time `wireSidebarButtons` runs; any class toggles, focus, or expando props set between builds are lost. Behavior looks intentional, just noting.
- `helpers/core.js:10` — `LUX_USER_ID` is computed at module load via `getUID()` and never refreshed. Outside this diff's scope, but since this file was touched today, worth flagging: any post-login identity rotation leaves this stale.
