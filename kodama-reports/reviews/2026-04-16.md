## `features/harvard/modal-actions.js`

### WARNING-1: `warnSwallow` assumed global, no fallback
- **Location:** `features/harvard/modal-actions.js:96, 133`
- **Issue:** The `catch` blocks call `globalThis.warnSwallow(...)` directly. `ui/lux-warn.js:85` only assigns `window.warnSwallow` (not `globalThis.warnSwallow`) — in most browsers these resolve to the same object so it works, but if this module runs before `lux-warn.js` has loaded, or in a context where `warnSwallow` is missing (SSR/test without the vi.fn stub), the catch itself throws a `TypeError`, masking the original error. Compare to the defensive guard in `features/results/header-modern.js:120` (`if (typeof globalThis.warnSwallow === "function")`). This file should use the same guard.
- **Confidence:** High. I verified both the global assignment site and the defensive pattern used elsewhere in the codebase.

### WARNING-2: Missing null guards on required DOM parameters
- **Location:** `features/harvard/modal-actions.js:61, 84, 91, 99, 122, 128, 149, 169, 239, 247`
- **Issue:** Functions like `selectHarvardList`, `selectPassage`, `switchTabCore`, and `showHoverHarvard` destructure a cloud of DOM-element parameters and then dereference them with no null check (e.g., `selTitle.textContent = rec.name` at line 84, `practiceBtn.textContent = ...` at line 91, `hoverTitle.textContent = rec.name` at line 149). The only guard is on `listEl` (line 73, 111). If a caller ever forgets a parameter — or if an element is removed from the DOM during a re-render — the function throws `Cannot set properties of null`. Given the number of optional-chained lookups elsewhere in these same functions (`phonTitleEl`, `filterClearBtn`, etc.), the mix of defensive and non-defensive access suggests an inconsistency rather than an intentional contract.
- **Confidence:** Medium — I'm not sure how tightly the callers are coupled. If every caller always passes every element, this is INFO; if any call site passes a subset, it's a live bug.

### INFO-3: `hideHover` identifier shadowed in `switchTabCore`
- **Location:** `features/harvard/modal-actions.js:176, 226, 255`
- **Issue:** The module exports a top-level function `hideHover` (line 176), and `switchTabCore` also accepts a `hideHover` parameter (line 226) that it calls at line 255 (`hideHover?.()`). Inside `switchTabCore`, the parameter shadows the export, so if a caller forgets to pass it, the `?.()` silently no-ops instead of falling back to the local export. Not a bug today, but a footgun.
- **Confidence:** High.

## `features/results/summary.js`

### WARNING-3: Unescaped user/ASR-derived text injected into HTML
- **Location:** `features/results/summary.js:158, 171`
- **Issue:** Line 158 interpolates `${err.word}` directly into an HTML string that is later assigned via `$out.innerHTML = html` (line 171). `err.word` originates from `nb.Words[].Word` — i.e., Azure Speech-to-Text output of whatever the user said. Azure's word field is generally whitespace-stripped alphanumerics, so this is unlikely to carry `<script>`, but:
  1. The `escapeHtml` helper is already imported elsewhere in the codebase (`ui/auth-dom.js:14`) and was *explicitly used* a few lines later in `auth-dom.js:118`, so the pattern is established.
  2. A word containing `&`, `<`, or `"` would also silently corrupt the rendered pill (and the `title="${fmtPctCefr(err.score)}"` quote attribute).
  3. Trust boundary: the STT response is technically external data, not pure user input, but it's still not a trusted HTML source.
- **Confidence:** High that this is an inconsistency with the rest of the codebase. Medium that it's actually exploitable.

### WARNING-4: Redundant `isCorrupt` filter masks the real intent

Either the guard on line 49 was added later and the filter is dead, or the filter is there because someone didn't trust the guard. Either way a reader is left guessing. (Not a bug — just adds confusion during debugging.)
- **Confidence:** High.

## `ui/auth-dom.js`

### WARNING-5: `session.user.email` assumed to exist
- **Location:** `ui/auth-dom.js:139–140`
- **Issue:**
  ```js
  const email = session.user.email;
  const name = email.split("@")[0];
  ```
  Supabase sessions can have `user.email === null` (phone auth, certain OAuth providers). The UI here is magic-link-only today, so this is unlikely to fire in production — but if anyone later enables another auth method, `email.split` will throw a `TypeError` and the entire `onAuthStateChange` callback aborts, which means `setUID(realUid)` and the dashboard refresh never run. Protect with `email?.split("@")[0] ?? "Account"`.
- **Confidence:** Medium. I'm not sure what auth methods are enabled on the Supabase project — if it's truly magic-link-only forever, this is INFO.

### WARNING-6: Race condition in `onAuthStateChange` async callback
- **Location:** `ui/auth-dom.js:133–175`
- **Issue:** `supabase.auth.onAuthStateChange(async (event, session) => { ... })` — Supabase may fire multiple events back-to-back (`INITIAL_SESSION`, `TOKEN_REFRESHED`, `SIGNED_IN`), and the handler does *not* gate on `event` type. Every fire of a logged-in state repeats the migration check:
  1. First fire reads `getString(K_IDENTITY_UID)` → guest UID still present (migration hasn't finished writing yet).
  2. Before `migrateHistory` resolves, a second fire reads the guest UID *again* and kicks off a duplicate migration request.
  The `/_api/migrate` endpoint needs to be idempotent to survive this; if it isn't, a user may get their history duplicated or lose rows. There's also no `setUID(realUid)` until after `await migrateHistory(...)`, so `K_IDENTITY_UID` is still the guest UID on the second fire.
- **Confidence:** Medium-high. I'm not sure about the migrate endpoint's idempotency; if the backend dedupes on `(guestUid, userUid)` this is a non-issue, otherwise it's a real bug.

### INFO-6: `supabase.auth.signOut()` not awaited before reload
- **Location:** `ui/auth-dom.js:150–151`
- **Issue:** `supabase.auth.signOut(); window.location.reload();` — `signOut` is async and may still be flushing the session-clear request when reload fires. In practice the reload usually wins and the post-reload `onAuthStateChange` fires with no session anyway, but this is race-prone. `await` it inside an `async` arrow, or use `.finally(() => window.location.reload())`.
- **Confidence:** High that this is a minor correctness/consistency issue; low that it causes a user-visible bug.
