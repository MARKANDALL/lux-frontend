## 1. `_api/alt-meaning.js`

```js
const ALT_MEANING_URL = `${API_BASE}/api/router?route=alt-meaning`;
export async function fetchAltMeanings(payload = {}) {
  return apiFetch(ALT_MEANING_URL, {
    method: "POST",
    body: JSON.stringify(payload || {}),
  });
}
```

### WARNING — URL captured at module load
`API_BASE` is evaluated once when the module is first imported. In vitest (with `vi.resetModules()` + dynamic re-import) this is fine, but in the real app this means any runtime reassignment of `globalThis.API_BASE` after first import will not affect this helper. The other new helpers in `_api/` follow the same pattern, so it is consistent with the rest of the codebase — flagging it as a latent risk, not a bug.
*Confidence:* medium. No evidence the app ever mutates `API_BASE` at runtime today.

### INFO — Redundant null guard
`payload = {}` already defaults `undefined` to `{}`; the additional `JSON.stringify(payload || {})` only adds protection against an explicit `null` (which would otherwise stringify to `"null"`). Either tighten the default to `payload ?? {}` or drop the inner guard — having both is code smell.

### INFO — Stale header comment
Line 1 reads `// api/alt-meaning.js`, but the file lives at `_api/alt-meaning.js` after the `api → _api` rename. Cosmetic.

## 2. `_api/util.test.js`

### WARNING — `x-admin-token` assertion is too loose
Line 54 and 147 use `expect(opts.headers["x-admin-token"]).toBeTruthy()`. Because `apiFetch` (util.js:97) reads `import.meta.env.VITE_ADMIN_TOKEN` **before** storage, a real env token can silently satisfy `toBeTruthy()` while the intended `"test-token-123"` from the mock is never exercised. Tighten to `toBe("test-token-123")` so a leaked env value causes a visible failure.

### INFO — FormData path never verifies the real header is attached at runtime
Line 79 asserts `Content-Type` is **absent** on a FormData POST, matching util.js:101-102. That is correct behavior (the browser adds the multipart boundary), but the test cannot prove the browser actually does so in Node. Acceptable for a unit test — noting for completeness.

### INFO — Missing coverage for `promptIfMissing: true`
`apiFetch` accepts `promptIfMissing` / `promptLabel`. The prompt branch in `getAdminToken` (util.js:57-64) is never exercised. Low priority; would require stubbing `window.prompt`.

### INFO — No test for non-ok blob/text paths
util.js:112-128 throws a `Request failed (${status})` error when `responseType` is `blob` or `text` and the response is not ok. Tests only cover the ok path.

## 3. `_api/identity.test.js`

### WARNING — Key strings hardcoded instead of imported
Tests hardcode `"LUX_USER_ID"` and `"lux_user_id"` (lines 57-58, 75, 84, 148). Source imports them as `K_IDENTITY_UID` / `K_IDENTITY_UID_LEGACY` / `K_IDENTITY_UID_ALIAS` from `app-core/lux-storage.js`. Today the values match (verified: `"LUX_USER_ID"`, `"lux_user_id"`, `"lux.uid"`), but if the registry renames a key the source keeps working while the test fails in a confusing way. Recommend importing the constants in the test.

### WARNING — Alias key (`lux.uid`) never exercised
Source has a 3-tier migration chain: `KEY` → alias (`"lux.uid"`) → legacy (`"lux_user_id"`). The test covers canonical and legacy reads but never the `ALIAS_KEY = "lux.uid"` recovery path — the middle tier of the migration ladder. A regression that broke alias fallback would go undetected.

### INFO — `setItem`-throwing path untested
`ensureUID` / `setUID` wrap localStorage writes in try/catch and call `warnSwallow` on failure (identity.js:62, 91, 122, 130, 134). No test simulates a storage quota exception.

### INFO — Fallback UUID generator untested
`makeUUID` has a hand-rolled fallback for environments lacking `crypto.randomUUID` (identity.js:28-34). Node 18+ has `crypto.randomUUID`, so the fallback branch is never hit in CI. Low priority.

## 4. `tests/scoring.test.js`

### WARNING — Author-flagged `TODO: verify` on a live assertion
Line 213-217:
```js
// TODO: verify this test — depends on globalThis.getSpeakingRate being unset.
t("deriveFallbackScores fluency fallback is null without getSpeakingRate", () => {
  const z = deriveFallbackScores({ NBest: [{ Words: [] }] });
  assert.equal(z.fluency, null);
});
```
A test that the author is not sure is correct is worse than no test — it creates false confidence. Either verify and remove the TODO, or delete the case until the behavior is pinned down.

### WARNING — Suspicious behavior cemented by tests (null → "urgent")
Lines 94-99 document that `Number(null) === 0` routes a `null` pronunciation score into the `< 60` **urgent** tier rather than the non-finite fallback (`"coach"`). The test acknowledges this with a comment but does not challenge it. Shipping `coachingLevel(null) === "urgent"` to a learner means "no score recorded yet" is visually indistinguishable from "terrible score." This is a latent UX bug in `core/scoring/index.js`, not the test. Recommend filing a source-side ticket to treat `null`/`undefined` as a separate no-data state before adopting these tests as the canonical contract.

### INFO — Framework inconsistency (`node:assert` vs. vitest)
`tests/*.test.js` use `node:assert/strict` with a custom `t()` runner; `_api/*.test.js` use vitest. Two runners means two different CI invocations, two different mocking stories, and duplicated harness code. Pick one.

### INFO — `completeness = 67` assumes `Math.round`
Line 209 expects `z.completeness === 67` for 2/3 (66.666…). That's `Math.round`; if source uses `Math.floor` / `toFixed`, the assertion silently drifts. Adding a one-line comment citing the rounding convention would harden the test.

### INFO — Edge-case coverage gaps
No test for `scoreClass(NaN)`, negative scores, or scores > 100. `fmtPct(Infinity)` / `fmtPct(-1)` also unexercised.
