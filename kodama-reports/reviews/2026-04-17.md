## `ui/auth-dom.js`

### 🔴 CRITICAL-1 — Migration endpoint does not use `API_BASE`; uses `_api/` source-directory prefix
- **Location:** `ui/auth-dom.js:191` inside `migrateHistory()`.
- **Issue:** The call is:
  ```js
  const data = await apiFetch("/_api/migrate", { ... });
  ```
  This is the **same class of bug** that was just fixed in `_api/alt-meaning.js` (commits `9b1a430`, `bb816d1`: "alt-meaning API path drift"). Two problems:
  1. **Wrong prefix.** `_api/` is the source-directory name of frontend helpers; it is not a real HTTP route. Backend routes are `/api/...` (see `_api/util.js` and the alt-meaning fix, which uses `${API_BASE}/api/router?route=alt-meaning`). Nothing in the codebase exposes `/_api/migrate` as an endpoint.
  2. **Missing `API_BASE`.** In production (`import.meta.env.DEV === false`), `API_BASE` resolves to `https://luxury-language-api.vercel.app`. Because the path starts with `/`, `fetch("/_api/migrate")` resolves against the **frontend origin** (e.g. the Vercel frontend), not the API backend. In production this request will 404 on the frontend host.
- **Impact:** Silent data loss. Guest history is never migrated on first sign-in. The `catch` block only logs to `console.error`, so users see no error and the button turns green as if everything worked. This likely already fails in production.
- **Severity:** CRITICAL — directly defeats the advertised "Save Your Progress" feature.
- **Suggested fix (before / after):**

  **Before** (line 189–198):
  ```js
  async function migrateHistory(guestUid, userUid) {
      try {
          const data = await apiFetch("/_api/migrate", {
              method: "POST",
              body: JSON.stringify({ guestUid, userUid }),
          });
          console.log("[Auth] Migration result:", data);
      } catch (e) {
          console.error("[Auth] Migration failed:", e);
      }
  }
  ```

  **After** (follow the alt-meaning.js convention — import `API_BASE` is already present on line 9):
  ```js
  async function migrateHistory(guestUid, userUid) {
      try {
          const data = await apiFetch(`${API_BASE}/api/router?route=migrate`, {
              method: "POST",
              body: JSON.stringify({ guestUid, userUid }),
          });
          console.log("[Auth] Migration result:", data);
      } catch (e) {
          console.error("[Auth] Migration failed:", e);
      }
  }
  ```
  *Note:* The exact route (`/api/router?route=migrate` vs. `/api/migrate`) should be confirmed against the backend router. I'm confident about the `API_BASE` prefix being required; less confident about whether the backend uses a dedicated path or the router-style query — verify before merging.

### ⚠️ WARNING-1 — Assumes `session.user.email` is always present
- **Location:** `ui/auth-dom.js:139–140`
  ```js
  const email = session.user.email;
  const name = email.split("@")[0];
  ```
- **Issue:** Supabase sessions from OAuth providers (phone auth, anonymous auth, some SSO) can have `email === null`. `null.split(...)` throws `TypeError` and the entire auth state handler aborts **before** `setUID(realUid)` and before migration. The user appears stuck as "guest" with a thrown exception in console.
- **Fix (suggestion):**
  ```js
  const email = session.user.email || "";
  const name = email ? email.split("@")[0] : "Account";
  ```
- **Confidence:** High that the bug exists; medium on whether any current auth flow hits it (magic-link always populates email — but this is the only auth flow today, so risk depends on future provider additions).

### ⚠️ WARNING-2 — `onAuthStateChange` re-runs migration on every token refresh
- **Location:** `ui/auth-dom.js:133–162`
- **Issue:** Supabase fires `onAuthStateChange` for `SIGNED_IN`, `TOKEN_REFRESHED`, `USER_UPDATED`, and `INITIAL_SESSION`. The code does not check `event`. The migration guard on line 159 is `guestUid !== realUid`, but `K_IDENTITY_UID` is only updated by `setUID(realUid)` on line 165 — which runs **after** `await migrateHistory(...)`. If a token refresh fires while the first migration is still in-flight, a second migration can launch against the same `guestUid`. The backend must be idempotent for this to be safe.
- **Fix:** gate on `event === "SIGNED_IN"`, or check whether `guestUid` still starts with the guest prefix before migrating, or set `K_IDENTITY_UID = realUid` **before** awaiting `migrateHistory`.
- **Confidence:** Medium — I did not trace `setUID` into `_api/index.js` to confirm whether it writes `K_IDENTITY_UID` synchronously. If `setUID` does update storage, duplicate migrations are still possible during the `await` window.

### ⚠️ WARNING-3 — `supabase.auth.signOut()` promise not awaited before reload
- **Location:** `ui/auth-dom.js:150–151`
  ```js
  supabase.auth.signOut();
  window.location.reload();
  ```
- **Issue:** `signOut()` is async. `window.location.reload()` fires immediately, potentially before Supabase clears its own storage. On reload the app may briefly see a stale session. Usually benign because Supabase reads a JWT that has already been revoked locally, but it is a race.
- **Fix:** `await supabase.auth.signOut(); window.location.reload();` inside the `async` onclick.
- **Confidence:** High that the race exists; low that it causes a visible user-facing bug.

### ⚠️ WARNING-4 — Email validation is a substring check
- **Location:** `ui/auth-dom.js:96` — `if (!email || !email.includes("@"))`.
- **Issue:** `@` alone is not an email. Strings like `@`, `a@`, `@b` pass. Supabase will return an error, which the code does handle via the `error` branch — so this is a UX issue, not a security issue.
- **Fix:** use a minimal regex (`/^[^\s@]+@[^\s@]+\.[^\s@]+$/`) or rely entirely on the `<input type="email">` native validation.
- **Confidence:** High.
