## `features/harvard/modal-actions.js`

### WARNING — Unguarded DOM access on required-but-undocumented args

Multiple exported helpers (`selectHarvardList`, `selectPassage`, `showHoverHarvard`, `showHoverPassage`, `switchTabCore`) destructure `selTitle`, `practiceBtn`, `hoverTitle`, `phonRows`, etc. from a single options bag. Some args are guarded (`if (!listEl) return;`, `if (phonTitleEl)`), but right after the guard the code calls e.g. `selTitle.textContent = rec.name` (line 84), `practiceBtn.textContent = "..."` (line 91), `hoverTitle.textContent = rec.name` (line 149), and `selTitle.textContent` again at lines 239/247 — with no null check. If a caller forgets one of these handles, the function throws partway through, leaving the modal in an inconsistent state (selection toggled but title unchanged).

Not strictly a bug today (callers presumably pass everything), but the inconsistent guarding makes future refactors fragile.

### WARNING — `positionHoverCard` math looks suspect (low confidence)

Lines 188–197:

```js
const cardRect = card.getBoundingClientRect();
const btnRect = btn.getBoundingClientRect();
const listRect = listEl.getBoundingClientRect();
const left = Math.max(12, listRect.right - cardRect.left + 12);
let top = Math.max(12, btnRect.top - cardRect.top - 6);
hoverCard.style.left = `${left}px`;
```

`hoverCard.style.left` is interpreted relative to the hover card's offset parent — almost certainly `card`. So `left` should be `(listRect.right - cardRect.left) + 12` (a card-local offset), which is what the code computes. That part is fine. But the `Math.max(12, …)` floor only matters when the list panel is to the *left* of the card's left edge, which shouldn't happen if the list is inside the card. **I'm not sure about this one** — the clamp is harmless when the geometry is sane, so flagging as code smell rather than bug.

### INFO — `globalThis.warnSwallow` is assumed to exist

Lines 96 and 133 swallow setString errors via `globalThis.warnSwallow(...)`. If that helper isn't installed before this module runs (e.g. in tests, in a stripped build, or if init order changes), the catch handler itself throws and replaces a benign storage error with a hard crash. Consider `globalThis.warnSwallow?.(...)`.

### INFO — `dataset.populated = "1"` is never reset

`ensurePhonemeOptions` skips repopulation when `focusSel.dataset.populated === "1"`. If the underlying phoneme set changes later in the session (new passage uploaded, etc.), the dropdown will be stale. Probably fine for now since options are static, but worth a comment.

## `features/results/summary.js`

### WARNING — `innerHTML` rendered with un-escaped speech-recognition output

Line 159 interpolates `err.word` directly into HTML:

```js
return `
  <span style="…">
     ${err.word}
     <span … title="${fmtPctCefr(err.score)}">(${err.score}%)</span>
     …
  </span>
`;
```

`err.word` originates from Azure's NBest results. In normal use it's plain ASCII, but the renderer treats this as trusted markup. If Azure ever returns a token containing `<`, `&`, or a quote (intentional or fuzzed input), it lands in `$out.innerHTML = html` (line 171) unsanitized. Other places in the codebase already use `escHtml`/`escapeHtml` from `helpers/escape-html.js`. Apply it here too.

### INFO — Hardcoded `"rainbow"` passage fallback

Line 36: `const passageKey = allPartsResults?.[0]?.passage_key || "rainbow";` — if the score payload is missing the key, the YouTube reference panel will show the Rainbow passage video regardless of what the user actually read. Better to render no video panel than the wrong one.

### INFO — Threshold prose drifts from `scoreClassCore`

Line 168 hardcodes the message *"No word-level scores below 60%"*, but the actual cutoff is whatever `scoreClassCore` returns as `score-bad`. If the threshold ever moves, the empty-state copy lies. Pull the number from a single source of truth.

## `ui/auth-dom.js`

### CRITICAL — `migrateHistory` POSTs to a same-origin URL, bypassing `API_BASE`

Line 191:

```js
const data = await apiFetch("/_api/migrate", {
    method: "POST",
    body: JSON.stringify({ guestUid, userUid }),
});
```

This is the **same drift bug** that was just patched in `_api/alt-meaning.js` (commits `bb816d1`, `9b1a430`, `699f55b`). In production the frontend is hosted on a different origin than `luxury-language-api.vercel.app`, so `/_api/migrate` either 404s on the static host or hits a non-existent route. Guest history will silently fail to migrate when users sign in — and because `migrateHistory` swallows the error in a `try/catch` that only `console.error`s (line 197), nothing surfaces to the user. After migration "fails", `setUID(realUid)` still runs (line 165), so the guest history is permanently orphaned under the old UID.

**Suggested fix (before/after):**

```js
// before
import { API_BASE, apiFetch } from "../_api/util.js";
// …
const data = await apiFetch("/_api/migrate", {
    method: "POST",
    body: JSON.stringify({ guestUid, userUid }),
});
```

```js
// after
import { API_BASE, apiFetch } from "../_api/util.js";
// …
const data = await apiFetch(`${API_BASE}/api/router?route=migrate`, {
    method: "POST",
    body: JSON.stringify({ guestUid, userUid }),
});
```

(Adjust the route name to whatever the backend actually exposes — the convention from `alt-meaning.js` is `/api/router?route=<name>`.) Also consider re-throwing inside `migrateHistory` so the caller can choose whether to delay `setUID` until migration confirms.

### WARNING — Migration can race on duplicate `onAuthStateChange` events

`supabase.auth.onAuthStateChange` fires for `INITIAL_SESSION`, `SIGNED_IN`, and `TOKEN_REFRESHED`. The migration block (lines 156–162) runs on every event whose `session?.user` is truthy and whose `guestUid !== realUid`. `setUID` does write `realUid` back to `K_IDENTITY_UID` (so the second event will short-circuit), **but** the writes happen *after* `await migrateHistory(...)`. If two events fire close together (e.g. INITIAL_SESSION + SIGNED_IN at app load), both callbacks read the stale guest UID and fire concurrent `POST /_api/migrate` requests. Worth either de-duping by event type, or moving `setUID` ahead of the await with a "migration in flight" guard.

### INFO — Email validation is `email.includes("@")`

Line 96. Permits `@`, `a@`, `@b`. Acceptable for a magic-link flow (Supabase will reject), but worth a slightly stricter regex so users get instant feedback instead of a roundtrip.
