## `ui/auth-dom.js`

Renders the auth button + modal, listens to Supabase auth events, and triggers a guest→user history migration. One high-impact routing bug and several edge cases.

### CRITICAL-1: `migrateHistory` does not use `API_BASE` — will 404 in prod
- **Where:** `ui/auth-dom.js:191`
- **Why it matters:** Every other `_api/*` helper prefixes `API_BASE` (see the alt-meaning fix that just shipped). In production, `API_BASE` = `https://luxury-language-api.vercel.app`, so a hard-coded same-origin path like `/_api/migrate` hits the Vite/Vercel frontend host, not the API. The request silently fails, `console.error` swallows it (see WARNING-3), and the user's guest history is lost even though `setUID(realUid)` still runs and flips them to the authenticated UID.
- **Confidence:** High on the routing break. Medium on the exact path — I did not find a `migrate` route in `_api/`, so the correct backend path should be confirmed against the API repo before patching (it may be `/api/migrate` or `/api/router?route=migrate`, matching alt-meaning's convention).

Before:
```js
// ui/auth-dom.js:189-198
async function migrateHistory(guestUid, userUid) {
    try {
        const data = await apiFetch("/_api/migrate", {
            method: "POST",
            body: JSON.stringify({ guestUid, userUid }),
        });
        console.log("[Auth] Migration result:", data);
    } catch (e) {
        console.error("[Auth] Migration failed:", e);
    }
}
```

After:
```js
// ui/auth-dom.js:189-198
import { API_BASE, apiFetch } from "../_api/util.js"; // already imported

async function migrateHistory(guestUid, userUid) {
    try {
        const data = await apiFetch(`${API_BASE}/api/migrate`, {
            method: "POST",
            body: JSON.stringify({ guestUid, userUid }),
        });
        console.log("[Auth] Migration result:", data);
        return data;
    } catch (e) {
        console.error("[Auth] Migration failed:", e);
        throw e; // let caller decide whether to proceed with setUID
    }
}
```

### WARNING-1: `email.split("@")[0]` crashes when Supabase session has no email
- **Where:** `ui/auth-dom.js:139-143`
- **Why:** `session.user.email` can be `null`/`undefined` for OAuth/phone providers, or when the email is not verified yet. `null.split("@")` throws, the listener unwinds, and the button never swaps to the signed-in state.
- **Suggested fix:** `const name = (email || session.user.phone || "account").split("@")[0];`
- **Confidence:** High that email can be null in Supabase; unsure whether this app ever allows non-magic-link providers today, so severity depends on that.

### WARNING-2: Dynamic `import(...)` fallback has no `.catch`
- **Where:** `ui/auth-dom.js:172-175`
- **Why:** If `features/dashboard/index.js` fails to resolve (module rename, chunk load error after a deploy while the user's tab was open), the promise rejects unhandled. Users will see the auth success flow complete but a console error with no recovery.
- **Suggested fix:** append `.catch((err) => console.warn("[Auth] Dashboard refresh failed:", err))`.

### WARNING-3: Migration failure is silently swallowed, then `setUID` still runs
- **Where:** `ui/auth-dom.js:156-165` interacting with `189-198`
- **Why:** If `migrateHistory` throws, it's caught and logged inside the helper. Control returns to the listener and `setUID(realUid)` unconditionally flips the global UID. The user is now "signed in" but their guest history was never copied — and because `setUID` persists `realUid` to `K_IDENTITY_UID`, the next auth event will see `guestUid === realUid` and skip migration forever. The guest data becomes orphaned.
- **Suggested direction:** either (a) re-throw from `migrateHistory` and `await` before `setUID` so a failed migration leaves the guest UID intact for retry, or (b) store the unmigrated guest UID under a separate key (e.g. `K_PENDING_MIGRATION_UID`) and retry on next load.
- **Confidence:** High on the orphaning mechanism; medium on the right fix — depends on backend idempotency of `/migrate`.

### INFO-2: Email validator accepts `"@"`, `"a@"`, `"@b"`
- **Where:** `ui/auth-dom.js:96` — `if(!email || !email.includes("@"))`
- Supabase will reject it server-side, but UX would be better with a minimal regex (`/^\S+@\S+\.\S+$/`) or `type=email` validity check.

### INFO-3: `window.location.reload()` races `supabase.auth.signOut()`
- **Where:** `ui/auth-dom.js:149-153`
- `signOut()` returns a promise; the reload fires immediately and can cancel the in-flight revoke call. The token still rotates server-side on expiry, but not atomically. Consider `await supabase.auth.signOut(); window.location.reload();`.

### INFO-4: Inline `onclick="..."` attribute on close button
- **Where:** `ui/auth-dom.js:121-124`
- Inconsistent with the rest of the file, which uses `element.onclick = ...`. Also brittle if a future CSP disallows inline handlers.

### INFO-5: Hardcoded production API host
- **Where:** `_api/util.js:4` — `const PROD_API = "https://luxury-language-api.vercel.app";`
- Not in the diff scope, but relevant to CRITICAL-1: a single source of truth for the API origin is already established, so the migration helper just needs to consume it.
