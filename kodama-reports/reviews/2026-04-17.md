## `features/harvard/modal-actions.js`

### WARNING — Partial null-guards in shared helpers
Several exported helpers accept many destructured DOM refs but only guard one before dereferencing the rest.

- `selectHarvardList` (lines 73–97): guards `listEl` only, then assumes `selTitle`, `practiceBtn`, `phonRows`, and the render callbacks all exist. If a caller ever omits them (e.g. a future page that reuses this helper), the function throws instead of no-opping.
- `selectPassage` (lines 111–134): same pattern.
- `showHoverHarvard` / `showHoverPassage` (lines 136–174): no null guards at all — will throw if `hoverTitle`/`hoverCard`/`renderLines` are missing.
- `switchTabCore` (lines 238–253): reads/writes `selTitle.textContent`, `practiceBtn.textContent`, `practiceBtn.disabled`, `listEl.setAttribute` without checking any of them.

Confidence: medium. All call sites in the current codebase probably pass every param, so the risk is future regressions rather than today's bug.

### WARNING — `globalThis.warnSwallow` inside catch blocks
Lines 96 and 133:
```js
} catch (err) { globalThis.warnSwallow("features/harvard/modal-actions.js", err, "important"); }
```
If `warnSwallow` isn't registered on `globalThis` (startup ordering, direct module import from a test), the catch handler itself throws `TypeError: globalThis.warnSwallow is not a function`, defeating the swallow. Same pattern appears in `_api/identity.js`, so this is a codebase-wide concern — but it was newly introduced on this file.

Confidence: high on the mechanics, low on whether it actually fires in practice (depends on bootstrap order).

### INFO — `positionHoverCard` math mixes coordinate frames
Line 193: `const left = Math.max(12, listRect.right - cardRect.left + 12);`
This subtracts the containing card's viewport-left from the list's viewport-right. That is only the right formula if `hoverCard` is absolutely positioned relative to `card`. If it's ever positioned relative to the viewport or document, the hover card will appear far from where you expect. A one-line comment pinning that assumption would help.

## `features/results/summary.js`

### WARNING — Unescaped user-adjacent strings in `innerHTML`
Line 158:
```js
return `
  <span style="...">
     ${err.word}
     <span ... title="${fmtPctCefr(err.score)}">(${err.score}%)</span>
     ...
  </span>
`;
```
`err.word` comes from `nb.Words[].Word` (Azure Speech) and is interpolated straight into HTML. `isCorrupt(w.Word)` filters out garbage but does **not** escape `<`, `>`, `&`, `"`. If a word ever contains those characters (contracted transcripts, unusual tokens, future backend changes), the markup breaks and we get an XSS-adjacent primitive. Use the existing `escapeHtml` helper (already imported in other files) or set `textContent`.

`title="${fmtPctCefr(err.score)}"` has the same concern inside an attribute, though `err.score` is numeric and currently safe.

Line 119 (`src="https://www.youtube.com/embed/${youtubeId}"`) is lower risk because `youtubeId` comes from the static `passages.js` data file, but an `encodeURIComponent` would harden it against a typo or future dynamic source.

Confidence: high that the escape is missing; medium that it will bite in production without a specific input.

### INFO — Silent fallback to `"rainbow"` passage key
Line 36:
```js
const passageKey = allPartsResults?.[0]?.passage_key || "rainbow";
```
If the results payload is missing `passage_key`, the UI renders the rainbow passage's video and parts anyway. That masks real data bugs. Prefer bailing out of the YouTube block and skipping the Voice Mirror mount when the key is missing.

### INFO — Inline-style duplication
Every block constructs colors from string literals. `getColorConfig` centralizes the pill/box colors for word scores, but the surrounding headings and "no word-level scores" banner still repeat hex codes. Not urgent.

## `ui/auth-dom.js`

### CRITICAL — `migrateHistory` POSTs to a non-existent URL
Lines 189–198:
```js
async function migrateHistory(guestUid, userUid) {
    try {
        const data = await apiFetch("/_api/migrate", {
            method: "POST",
            body: JSON.stringify({ guestUid, userUid }),
        });
        ...
    } catch (e) { console.error("[Auth] Migration failed:", e); }
}
```

Three problems stacked:
1. The URL is `/_api/migrate`. The rest of the frontend calls `${API_BASE}/api/...`. `API_BASE` is imported at line 9 but never used here.
2. `vite.config.js` only proxies `/api` (not `/_api`) in dev, so this 404s during local development.
3. Vercel treats top-level folders with a leading underscore as private, so `_api/` is not served as functions in production; `/_api/migrate` will hit the static frontend and return 404/HTML, then `apiFetch` will throw "Bad JSON".

Because the catch just `console.error`s, the failure is invisible to users — login appears to succeed, but **no guest-to-user history migration ever happens**.

**Suggested fix** (before → after):

Before:
```js
async function migrateHistory(guestUid, userUid) {
    try {
        const data = await apiFetch("/_api/migrate", {
            method: "POST",
            body: JSON.stringify({ guestUid, userUid }),
        });
        console.log("[Auth] Migration result:", data);
    } catch (e) {
        console.error("[Auth] Migration failed:", e);
    }
}
```

After:
```js
async function migrateHistory(guestUid, userUid) {
    try {
        const data = await apiFetch(`${API_BASE}/api/router?route=migrate`, {
            method: "POST",
            body: JSON.stringify({ guestUid, userUid }),
        });
        console.log("[Auth] Migration result:", data);
        return data;
    } catch (e) {
        console.error("[Auth] Migration failed:", e);
        throw e; // let caller decide whether to continue
    }
}
```

Notes for the fix:
- I'm **not certain** the correct prod route is `route=migrate` — the reviewer should confirm against the backend's `luxury-language-api` router. The relevant fact is that it must go through `${API_BASE}/api/...`, not `/_api/...`.
- Consider surfacing the error rather than silently logging, so failed migrations don't masquerade as successful logins.

Confidence: high that the current URL is wrong; medium on the exact replacement route name.

### WARNING — Migration can run more than once per sign-in
Lines 155–165:
```js
const guestUid = getString(K_IDENTITY_UID);
if (guestUid && guestUid !== realUid) {
    await migrateHistory(guestUid, realUid);
}
setUID(realUid);
```

`supabase.auth.onAuthStateChange` fires for `SIGNED_IN`, `TOKEN_REFRESHED`, and `USER_UPDATED`. If `TOKEN_REFRESHED` (or a second tab) fires while `migrateHistory(...)` is still awaiting, both handlers see the same `guestUid` (setUID hasn't run yet) and both trigger a POST. Backend idempotency may save us, but we shouldn't rely on it.

Mitigation: set a module-level `migrationInFlight` promise, or call `setUID(realUid)` *before* awaiting the network, or gate on an `in_progress` sentinel in storage.

Confidence: medium — depends on how aggressively Supabase re-emits `TOKEN_REFRESHED` and whether the backend is idempotent.

### WARNING — `session.user.email` assumed non-null
Line 140:
```js
const email = session.user.email;
const name = email.split("@")[0];
```
Supabase users created via phone or anonymous sign-in have `email === null`. `.split` then throws and the whole handler aborts before `setUID`/dashboard refresh run. Use `session.user.email ?? ""` or guard with `if (email)`.

Confidence: high on the null case existing, low on whether this app ever uses non-email auth.
