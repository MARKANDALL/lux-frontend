## HIGH severity findings

### 1. `vite` — current `6.4.1` (DIRECT dev dependency)

- **What:** Path traversal in optimized-deps `.map` handling + arbitrary file read via dev-server WebSocket (GHSA-p9ff-h696-f583, GHSA-4w7w-66w2-5vf9). An attacker who can reach the dev server can read files outside the project root.
- **Used in Lux:** `vite.config.js:4` imports `defineConfig, loadEnv` from `"vite"`. Build + dev scripts (`npm run dev`, `npm run build`, `npm run preview`).
- **Exposure:** **Dev-time only.** Not shipped to users. Risk realised only if a dev runs the dev server on a reachable network (e.g. `--host`, codesandbox).
- **Safe upgrade exists:** Yes — `6.4.2` (patch bump within current major `^6.0.0`).
- **Fix command (report only, do not run):**
  ```
  npm install --save-dev vite@^6.4.2
  ```
- **Risk of upgrading:** **Low.** Patch version bump within the `^6` range already declared; no breaking changes expected.

### 2. `rollup` — current `4.53.3` (transitive via `vite`)

- **What:** Arbitrary file write via path traversal in Rollup 4 (GHSA-mw96-cpmx-2vgc). Malicious build inputs could write outside the build output dir.
- **Used in Lux:** Not imported directly. Pulled in by `vite` for bundling (`vite.config.js` uses `build.rollupOptions`).
- **Exposure:** **Build-time only.** Triggered only if a malicious plugin/input is fed to Rollup; very low in a private codebase.
- **Safe upgrade exists:** Yes — `4.59.0+` (`4.60.1` latest). Fixed by updating Vite (Rollup is a transitive).
- **Fix command:**
  ```
  npm update rollup            # or simply update vite; rollup will ride along
  ```
- **Risk of upgrading:** **Low.** Minor version bump. Vite 6.4.2 pins a compatible Rollup.

### 3. `picomatch` — current `4.0.3` (transitive via `vite` / `vitest`)

- **What:** ReDoS via extglob quantifiers + method injection in POSIX character classes (GHSA-c2c7-rcm5-vvqj, GHSA-3v7f-55p6-f55p). A crafted glob pattern can hang the process or match incorrectly.
- **Used in Lux:** Not imported directly. Used internally by Vite and Vitest for file globbing.
- **Exposure:** **Dev/test-time only.** Patterns are supplied by our config, not by untrusted input.
- **Safe upgrade exists:** Yes — `4.0.4` (patch).
- **Fix command:**
  ```
  npm update picomatch
  ```
- **Risk of upgrading:** **Low.** Patch within the `^4` range declared by Vite/Vitest.

### 4. `flatted` — current `3.3.3` (transitive via `eslint` → `keyv` → `flat-cache`)

- **What:** Unbounded-recursion DoS + prototype pollution in `parse()` (GHSA-25h7-pfq9-p65f, GHSA-rf6f-7fwh-wjgh). Crafted JSON fed to `flatted.parse()` can crash the process or pollute `Object.prototype`.
- **Used in Lux:** Not imported directly. ESLint uses it to serialize its cache file (`.eslintcache`).
- **Exposure:** **Dev-time only**, and only if an attacker can write to the local `.eslintcache` file (already a local-box compromise).
- **Safe upgrade exists:** Yes — `3.4.2` (minor).
- **Fix command:**
  ```
  npm update flatted
  ```
- **Risk of upgrading:** **Low.** Minor version bump.

### 5. `minimatch` — current `3.1.2` (transitive via `eslint`, `@humanwhocodes/config-array`, `glob`)

- **What:** Three separate ReDoS advisories (GHSA-3ppc-4f35-3m26, GHSA-7r86-cg39-jmmj, GHSA-23c5-xmqv-rm74). Crafted glob patterns can trigger catastrophic backtracking.
- **Used in Lux:** Not imported directly. Pulled in by the legacy `eslint@8` dependency tree.
- **Exposure:** **Dev-time only.** Patterns are supplied by our lint config.
- **Safe upgrade exists:** Yes in the `3.x` series — `3.1.4+` (`3.1.5` latest v3). The latest overall is `10.2.5` (major jump, not compatible with eslint 8's transitive pins).
- **Fix command:**
  ```
  npm update minimatch        # will bump to the highest semver-compatible 3.1.x
  ```
  For a fuller fix, migrate off the legacy ESLint 8 tree (see Maintenance Risk below).
- **Risk of upgrading:** **Low** for the `3.1.x` patch. **Medium-High** to follow it up by upgrading to ESLint 9 (flat-config migration required).

### Notes

- **`cmu-pronouncing-dictionary`** has not been published in almost 4 years. It's a static data package (CMU pronunciation dictionary) rather than executable logic, so the maintenance risk is moderate — a data package that rarely needs updates is not the same as an unmaintained code library. Still worth a manual review: confirm the source data is acceptable for production and that there is no active fork we should track.
- **`eslint@^8.57.1`** is maintained but the major line is superseded by ESLint 9 (flat config). Staying on 8 keeps pulling in legacy `minimatch@3` / `ajv@6` transitives; migrating to ESLint 9 would clear multiple transitive advisories in one move but requires a flat-config port (`.eslintrc.json` → `eslint.config.js`).

## Recommended order of operations (report only — do not apply in this PR)

1. **Patch Vite 6.4.1 → 6.4.2.** Closes the direct HIGH advisory and also pulls fixed Rollup and picomatch transitively. Lowest-risk change.
2. **`npm update`** on `flatted`, `minimatch`, `picomatch`, `rollup` to backfill any transitives that Vite's update doesn't reach.
3. **Re-run `npm audit`** — moderate issues in `ajv` / `brace-expansion` should also clear because they share the ESLint dep tree.
4. **Plan (separate effort)** an ESLint 8 → 9 migration to eliminate the legacy minimatch/ajv subtree at its source.
5. **Investigate** `cmu-pronouncing-dictionary` staleness — decide between pinning, forking, or replacing.

## Verdict

Not clean, but **not urgent for end users**. Every HIGH finding is in a dev/build-only code path, and every one has a safe, low-risk patch available. Production runtime dependencies (`@supabase/supabase-js`, `cmu-pronouncing-dictionary`) are unaffected by these advisories.
