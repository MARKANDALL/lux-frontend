## HIGH-severity vulnerabilities

### 1. `vite` — HIGH (direct dev dependency)

- Installed range: `^6.0.0` (vulnerable: `<=6.4.1`)
- Fixed in: `6.4.2` (patch within `^6`), `7.x`, `8.0.9` (latest)
- Advisories:
  - [GHSA-p9ff-h696-f583](https://github.com/advisories/GHSA-p9ff-h696-f583) — HIGH — Arbitrary File Read via Vite Dev Server WebSocket
  - [GHSA-4w7w-66w2-5vf9](https://github.com/advisories/GHSA-4w7w-66w2-5vf9) — MODERATE — Path Traversal in Optimized Deps `.map` Handling
- Plain English: A malicious web page loaded in a browser that also has the
  Vite dev server open can trick the dev server into returning arbitrary
  files from the developer's machine over WebSocket.
- Lux usage: `vite.config.js:4` imports `defineConfig, loadEnv` from `vite`.
  Used for `npm run dev` / `npm run build` / `npm run preview`. Not shipped to
  production.
- Suggested fix (do not run — report only):
  - Safest (in-range patch): edit `package.json` → `"vite": "^6.4.2"`, then
    `npm install`.
  - Latest: `npm install --save-dev vite@^8` — but Vite 7 and 8 both have
    breaking config/plugin changes vs. Vite 6; regression-test the dev server,
    HMR, and `vite build` output before merging.
- Risk of upgrade:
  - **Low** for `6.4.1 → 6.4.2` (patch bump — safe).
  - **High** for `6 → 8` (two major versions; Rollup 4 API changes inside Vite,
    plugin API shifts). Would require a dedicated branch + manual QA.

### 2. `rollup` — HIGH (transitive, via `vite`)

- Installed range: `4.0.0 - 4.58.0` (vulnerable)
- Fixed in: `4.59.0` (latest `4.60.2`)
- Advisory: [GHSA-mw96-cpmx-2vgc](https://github.com/advisories/GHSA-mw96-cpmx-2vgc) — Arbitrary File Write via Path Traversal in Rollup 4
- Plain English: Rollup 4 can be coerced into writing output files outside
  the intended `outDir` if a plugin or input config contains attacker-
  controlled path strings. Relevant to any CI that consumes untrusted plugin
  input — not a typical Lux build scenario, but still a real hole.
- Lux usage: **No direct imports.** Pulled in transitively by `vite`. Search
  for `from "rollup"` / `require("rollup")` returns zero hits in Lux source
  (the `rollup`/`rollups` matches in `features/progress/rollups.js` etc. are
  domain-level filename collisions, not imports).
- Suggested fix (do not run — report only):
  - Bumping `vite` to `6.4.2+` should carry a newer `rollup` automatically
    via its dependency range. If not, `npm update rollup` or a `package.json`
    `overrides` block pinning `rollup` to `^4.59.0`.
- Risk of upgrade: **Low.** Patch/minor within Rollup 4.x line — API stable.

### 3. `minimatch` — HIGH (transitive, via `eslint`, `glob`)

- Installed range: `<=3.1.3` (vulnerable)
- Fixed in: `3.1.4`+ (latest `10.2.5`)
- Advisories (all ReDoS):
  - [GHSA-3ppc-4f35-3m26](https://github.com/advisories/GHSA-3ppc-4f35-3m26)
  - [GHSA-7r86-cg39-jmmj](https://github.com/advisories/GHSA-7r86-cg39-jmmj)
  - [GHSA-23c5-xmqv-rm74](https://github.com/advisories/GHSA-23c5-xmqv-rm74)
- Plain English: Crafted glob patterns (e.g. nested `*()` extglobs, multiple
  `**` segments) cause catastrophic regex backtracking and pin a CPU core
  indefinitely. Only exploitable if untrusted input reaches a `minimatch`
  call.
- Lux usage: **No direct imports.** Transitive via `@eslint/eslintrc`,
  `@humanwhocodes/config-array`, `eslint`, `glob`. All dev-only.
- Suggested fix (do not run — report only):
  - `npm audit fix` will hoist `minimatch@3.1.4+`. Alternatively a `package.json`
    `overrides: { "minimatch": "^3.1.4" }` block resolves it without an eslint
    major bump.
- Risk of upgrade: **Low.** Patch within `3.x` line. Eslint 8 is compatible.

### 4. `picomatch` — HIGH (transitive, via `vite` → `chokidar`/`fdir`)

- Installed range: `4.0.0 - 4.0.3` (vulnerable)
- Fixed in: `4.0.4` (latest)
- Advisories:
  - [GHSA-c2c7-rcm5-vvqj](https://github.com/advisories/GHSA-c2c7-rcm5-vvqj) — HIGH — ReDoS via extglob quantifiers
  - [GHSA-3v7f-55p6-f55p](https://github.com/advisories/GHSA-3v7f-55p6-f55p) — MODERATE — Method Injection in POSIX Character Classes
- Plain English: Same class of ReDoS as `minimatch` — crafted glob patterns
  kill the process. Impact here is low because pattern strings come from the
  Lux repo itself (our own vite config), not user input.
- Lux usage: **No direct imports.** Transitive via `vite`.
- Suggested fix (do not run — report only):
  - `npm update picomatch` to pick up `4.0.4`. Pure patch bump.
- Risk of upgrade: **Low.** Patch version, no API changes.

### 5. `flatted` — HIGH (transitive, via `eslint` → `@eslint/eslintrc`)

- Installed range: `<=3.4.1` (vulnerable)
- Fixed in: `3.4.2`
- Advisories:
  - [GHSA-25h7-pfq9-p65f](https://github.com/advisories/GHSA-25h7-pfq9-p65f) — HIGH — Unbounded recursion DoS in `parse()` revive phase
  - [GHSA-rf6f-7fwh-wjgh](https://github.com/advisories/GHSA-rf6f-7fwh-wjgh) — HIGH — Prototype Pollution via `parse()`
- Plain English: Parsing a deeply-nested or hostile JSON-like payload with
  `flatted.parse()` either crashes the process (stack overflow) or pollutes
  `Object.prototype`. Dangerous only if untrusted payloads reach
  `flatted.parse`. Inside Lux, `flatted` is only used by eslint's lint cache
  — lint input is your own source code, not attacker input.
- Lux usage: **No direct imports.** Transitive via eslint's lint cache.
- Suggested fix (do not run — report only):
  - `npm update flatted` or `overrides: { "flatted": "^3.4.2" }`.
- Risk of upgrade: **Low.** Patch version.
