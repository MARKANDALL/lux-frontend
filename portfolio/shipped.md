### Notes for future Mark
The routines fleet looks more polished than it is. The registry rename (`a333aa0` + `5aeeaa4`) was a single afternoon's work, and Utility Gate v2 is one `cat`/`diff` against a state file — not a deep idempotency system. Don't oversell it as "distributed job orchestration." It's a disciplined convention on top of cron, and the honest interview framing is "I noticed I was re-running expensive scans on unchanged state and added a two-line gate to fix it" — which is still a good answer, just not a bigger one than it is.

The docs rewrite was partly panic. The pillar-symmetry framing got invented *during* the rewrite, not before it — the extraction-ledger idea emerged because Mark didn't trust himself to remember what was being moved. That's fine and arguably better (shipped first, formalized second), but don't retrofit the narrative into "I designed a docs system." The truthful story is "I rewrote the docs, and in doing so discovered the invariants they should have had."

Test-import fix (PR #31) was 27 lines of find-replace. The interview value is the *diagnosis* — noticing that the test suite had been silently broken for weeks — not the fix. Lead with the diagnosis.

No production incidents this week. No customer-story-shaped narratives. Work was mostly internal tooling and docs. That's honest and still interview-usable, but save "diagnosed a prod bug" anecdotes for when they actually happen.
